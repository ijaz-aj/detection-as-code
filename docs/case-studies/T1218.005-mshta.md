# Case Study — T1218.005 Mshta Proxy Execution

**Rule:** [`lolbin-mshta-rundll32`](../../detections/defense-evasion/lolbin-mshta-rundll32/)
**Tactic / Technique:** Defense Evasion / T1218.005
**Date:** 2026-07-07

## 1. Attack (Atomic Red Team)

```powershell
Invoke-AtomicTest T1218.005 -TestNumbers 2   # Mshta executes VBScript (FIN7 emulation)
```
Command run:
```
mshta vbscript:Execute("CreateObject(""Wscript.Shell"").Run ""powershell -noexit -file C:\AtomicRedTeam\atomics\T1218.005\src\powershell.ps1"":close")
```
It uses `mshta.exe` to run inline VBScript, which launches a local PowerShell user-enumeration
script — emulating how FIN7 abuses mshta.

## 2. Obstacle — Windows Defender blocked it (a real finding)

The first run failed with `Access is denied`. Windows Defender **real-time protection** blocked
`mshta.exe` running inline VBScript to spawn PowerShell — a textbook malicious pattern. The
`C:\AtomicRedTeam` folder exclusion did **not** cover this: `mshta.exe` lives in `system32` and the
VBScript is inline (not a file in the excluded folder). Disabled real-time protection on the
**isolated lab VM** to test the detection.

> Note: Defender blocking this by default is itself a positive defensive signal — the technique is
> so consistently malicious that the built-in AV stops it.

## 3. Detect — clean catch ✅

Hunted in Kibana Discover:
```
event.code : "1" and process.name : "mshta.exe"
```
The `mshta.exe` event's `process.command_line` contained `vbscript:Execute(...)`. Pasting Rule 3's
Elastic (Lucene) query matched it on the **first try — no tuning needed**. The literal string
`vbscript` matches case-consistently, unlike the T1059.001 case-sensitivity trap.

## 4. Contrast with T1059.001

- **Rule 1 (encoded PowerShell):** MISS → diagnose → tune → catch (Elastic case-sensitivity).
- **Rule 3 (mshta):** clean catch, first try.

Together these show detection range: recognising when a rule needs tuning versus when a
well-scoped rule simply works.
