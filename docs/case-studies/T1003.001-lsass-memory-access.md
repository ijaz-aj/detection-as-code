# Case Study — T1003.001 LSASS Memory Access (Credential Dumping)

**Rule:** [`lsass-memory-access`](../../detections/credential-access/lsass-memory-access/)
**Tactic / Technique:** Credential Access / T1003.001
**Date:** 2026-07-07

> The richest loop of the project: one attack surfaced **three** distinct findings —
> a telemetry gap, a precision problem, and a recall problem.

## 1. Attack (Atomic Red Team)

```powershell
Invoke-AtomicTest T1003.001 -TestNumbers 2   # Dump LSASS.exe Memory using comsvcs.dll
```
A living-off-the-land credential dump: `rundll32.exe comsvcs.dll, MiniDump <lsass_pid> dump.bin full`
uses a **built-in Windows DLL** to write LSASS memory (containing credentials) to a file — no external
tooling. Ties back to the Rule 3 rundll32/LOLBin technique.

## 2. Finding #1 — Sensor gap: the telemetry wasn't being collected

Before attacking, checked whether the required telemetry even exists:
```
event.code : "10" and winlog.channel : "Microsoft-Windows-Sysmon/Operational"   -> ZERO results
```
Rule 2 depends on Sysmon **Event ID 10 (ProcessAccess)**, but the SwiftOnSecurity config **disables it
by default** (it is high-volume). The config says so directly:
```xml
<!--SYSMON EVENT ID 10 : INTER-PROCESS ACCESS [ProcessAccess]-->
    <!--COMMENT: Can cause high system load, disabled by default.-->
    <ProcessAccess onmatch="include">
        <!--NOTE: Using "include" with no rules means nothing in this section will be logged-->
    </ProcessAccess>
```
**The rule was structurally blind — no rule change could have helped.** Fixed the *sensor*: enabled
ProcessAccess logging for LSASS and reloaded Sysmon live.
```xml
<ProcessAccess onmatch="include">
    <TargetImage name="T1003.001,LSASS" condition="image">lsass.exe</TargetImage>
</ProcessAccess>
```
```powershell
C:\Sysmon\Sysmon64.exe -c C:\Sysmon\sysmonconfig.xml   # live reload
```
After the reload, EID 10 events for LSASS began flowing. **Detection depends on telemetry, and
telemetry depends on sensor configuration** — the #1 lesson of the loop.

## 3. Detect — the attack IS caught, but with two problems

The comsvcs dump opened LSASS **twice**, producing two EID 10 events from `rundll32.exe`:

| GrantedAccess | Meaning | In original rule? |
|---|---|---|
| `0x1410` | query info + `PROCESS_VM_READ` (read memory) | ✅ yes → **caught** |
| `0x1fffff` | `PROCESS_ALL_ACCESS` (every right) | ❌ no → **missed** |

**GrantedAccess** is the bitmask of rights a process was granted when it opened LSASS. Credential
dumping requires reading memory (`PROCESS_VM_READ`, `0x0010`), which is why the rule keys on masks
containing that bit. The attack was detected via `0x1410` — but two issues showed up.

### Finding #2 — Precision: the monitoring agent floods the rule
Running the rule returned **136 hits**, almost all from `...\elastic-agent\...\agentbeat.exe` reading
LSASS with `0x1010` as part of normal process-metadata collection. **The security agent looks like a
credential dumper.** The original `filter_legit` excluded wininit/csrss/wmiprvse/Defender, but not the
Elastic Agent.

### Finding #3 — Recall: the full-access dump mask was missing
The `0x1fffff` (full access) open was *not* in the rule's mask list, so that event was missed.
`0x1fffff` is a common dumping mask (ProcDump `-ma`, etc.) — a tool using only full access would evade.

## 4. Tune

Two edits to the rule:
```yaml
    GrantedAccess:
      - ...
      - '0x1fffff'                       # ADD: full access - recall
  filter_agent:
    SourceImage|contains: '\Elastic\Agent\'   # ADD: exclude the agent by PATH - precision
  condition: selection and not 1 of filter_*
```
(Excluded the agent by **path**, not exe name, so an attacker can't evade by renaming their tool
`agentbeat.exe`.)

## 5. Verify — clean catch

Re-running the tuned query on live data:

| | Result |
|---|---|
| **Before** | 136 hits (mostly `agentbeat.exe` false positives) |
| **After** | **2 hits — both the real `rundll32 → lsass` dump** (`0x1410` *and* `0x1fffff`) |

Precision *and* recall improved in one edit: agent noise eliminated, full-access dump now caught.
